# Практическая работа 6. Итоговая настройка release-версии проекта и запуск в Docker

Целью итоговой практической работы являлась подготовка проекта интернет-магазина футбольных товаров к запуску в release-режиме. В ходе работы нужно было привести структуру проекта в готовый для развёртывания вид, настроить запуск Django через Gunicorn, подключить Nginx как внешний веб-сервер, развернуть приложение в Docker и проверить, что сайт и административная панель открываются корректно.

Тема индивидуального проекта — интернет-магазин футбольных товаров «Футбол Маркет». Проект предназначен для просмотра каталога товаров, работы с категориями, клубами, брендами и поставщиками, добавления товаров в корзину, оформления заказов авторизованным пользователем, а также для управления данными через административную панель Django и REST API.

Описание предметной области. Интернет-магазин футбольных товаров представляет собой систему, в которой покупатель может просматривать товары, относящиеся к разным категориям: футбольная форма, мячи, бутсы, шарфы и другие аксессуары. Каждый товар связан с футбольным клубом, брендом и поставщиком, что позволяет хранить не только название и цену товара, но и дополнительную информацию о его происхождении и принадлежности. Покупатель может открыть карточку товара, посмотреть описание, цену и остаток на складе, после чего добавить товар в корзину. Корзина хранится в сессии пользователя, поэтому данные сохраняются между переходами по страницам. После авторизации пользователь может оформить заказ, указав адрес доставки. При создании заказа к нему автоматически привязывается пользователь, а позиции заказа сохраняют товар, количество и цену на момент оформления. Администратор магазина управляет справочниками, товарами, покупателями, заказами и позициями заказов через административную панель. Такая структура подходит для учебного проекта, потому что в ней есть основные сущности интернет-магазина, связи между таблицами, CRUD-операции, авторизация, корзина, API и отдельная release-настройка.

В начале работы была проверена структура проекта. Основное приложение находится в папке `football_shop`, API вынесено в отдельную папку `shop_api`, статические файлы расположены в папке `static/shop`, шаблоны расположены в папке `templates/shop`, настройки проекта находятся в `shop_project/settings.py`, маршруты проекта подключены через `shop_project/urls.py`, а запуск WSGI-приложения выполняется через `shop_project/wsgi.py`. Для release-запуска были добавлены файлы `Dockerfile`, `docker-compose.yml`, `.dockerignore`, `docker/entrypoint.sh` и конфигурация `docker/nginx/default.conf`.

Скриншот 1 — структура проекта после добавления Docker-файлов. На скриншоте необходимо показать, что в проекте появились файлы `Dockerfile`, `docker-compose.yml`, папка `docker`, а внутри неё находятся `entrypoint.sh` и конфигурация Nginx. Этот скриншот подтверждает, что структура проекта подготовлена для контейнерного запуска.

Для настройки release-версии были изменены настройки Django. Значения `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS`, `CSRF_TRUSTED_ORIGINS` и параметры подключения к PostgreSQL теперь читаются из переменных окружения через `python-decouple`. Это позволяет использовать одни и те же настройки в локальной разработке и внутри Docker, меняя только переменные окружения. Также был добавлен `STATIC_ROOT`, чтобы команда `collectstatic` могла собрать статические файлы в отдельную папку для отдачи через Nginx.

Скриншот 2 — фрагмент файла `shop_project/settings.py` с настройками `DEBUG`, `ALLOWED_HOSTS`, базы данных и `STATIC_ROOT`. На этом скриншоте нужно показать, что параметры проекта не зашиты жёстко в код, а берутся из переменных окружения.

Для запуска приложения был настроен Dockerfile. В образ устанавливается Python, системные зависимости для PostgreSQL-драйвера, зависимости из `requirements.txt`, затем копируется код проекта. После сборки запускается `entrypoint.sh`, который ждёт доступности PostgreSQL, выполняет миграции, собирает статические файлы и передаёт управление Gunicorn. Gunicorn запускает Django-приложение через `shop_project.wsgi:application` и слушает порт `8000` внутри контейнера.

Скриншот 3 — содержимое `Dockerfile` и `docker/entrypoint.sh`. Этот скриншот показывает, что проект запускается не через встроенный сервер разработки Django, а через Gunicorn, а миграции и сборка статики выполняются автоматически при старте контейнера.

Для объединения сервисов был создан `docker-compose.yml`. В нём описаны три сервиса: `db`, `web` и `nginx`. Сервис `db` использует PostgreSQL 15, сервис `web` собирает Django-приложение из Dockerfile, а сервис `nginx` принимает запросы на порту 80 и проксирует их в контейнер `web`. Также в compose-файле настроены volumes для хранения данных PostgreSQL и собранной статики.

Скриншот 4 — файл `docker-compose.yml`. На скриншоте нужно показать три сервиса: PostgreSQL, Django/Gunicorn и Nginx. Это подтверждает, что проект разворачивается не одним процессом, а полноценной связкой контейнеров.

Nginx был настроен как внешний веб-сервер. В файле `docker/nginx/default.conf` создан upstream `django_app`, который указывает на контейнер `web:8000`. Запросы к `/static/` отдаются напрямую из папки собранной статики, а остальные запросы проксируются в Gunicorn. Благодаря этому административная панель и страницы сайта получают CSS и JS через Nginx.

Скриншот 5 — конфигурация `docker/nginx/default.conf`. На нём нужно показать блок `location /static/` и блок `location /`, где используется `proxy_pass http://django_app`.

После настройки проект был запущен командой `docker compose up --build -d`. Во время запуска Docker собрал образ приложения, поднял контейнер PostgreSQL, дождался его готовности, применил миграции, собрал 174 статических файла и запустил Gunicorn. В логах web-контейнера видно сообщение `Starting gunicorn 23.0.0` и строку `Listening at: http://0.0.0.0:8000`.

Скриншот 6 — терминал с результатом команды `docker compose up --build -d` или логами `docker compose logs web`. На скриншоте должны быть видны применённые миграции, сборка статики и запуск Gunicorn.

Проверка контейнеров выполнялась командой `docker compose ps`. В результате были запущены три контейнера: `shop_project-db-1`, `shop_project-web-1` и `shop_project-nginx-1`. Контейнер базы данных получил статус `healthy`, Nginx был доступен на порту 80, а web-контейнер работал через entrypoint и Gunicorn.

Это подтверждает, что запрос проходит через Nginx к Django-приложению.

Логическая схема базы данных включает основные сущности предметной области: категория, футбольный клуб, бренд, поставщик, товар, покупатель, пользователь, заказ и позиция заказа. Категория, клуб, бренд и поставщик связаны с товарами отношением «один ко многим»: один бренд или одна категория могут относиться ко многим товарам. Заказ связан с покупателем и пользователем, а позиции заказа связывают заказ с конкретными товарами. Такая схема позволяет хранить справочники отдельно от товаров и фиксировать состав каждого заказа.

Физическая схема базы данных реализована через модели Django. Таблица `firstproject_product` содержит внешние ключи на `firstproject_category`, `firstproject_club`, `firstproject_brand` и `firstproject_supplier`. Таблица `firstproject_order` содержит внешние ключи на `firstproject_customer` и `auth_user`. Таблица `firstproject_orderitem` содержит внешние ключи на `firstproject_order` и `firstproject_product`. Для удаления связанных справочников используется `PROTECT`, чтобы нельзя было случайно удалить категорию, бренд, клуб, поставщика или товар, если они используются в связанных данных.

Скриншот 11 — актуальная логическая или физическая диаграмма базы данных. Здесь можно вставить схему, которую делали ранее, но обновить её с учётом таблицы пользователей и связи заказа с пользователем.

Словарь данных проекта включает следующие основные таблицы. `Category` хранит название, слаг и описание категории. `Club` хранит название футбольного клуба, страну, лигу, логотип и описание. `Brand` хранит название бренда, страну, логотип и описание. `Supplier` хранит название поставщика, город, email, телефон и описание. `Product` хранит название товара, слаг, категорию, клуб, бренд, поставщика, цену, остаток, фото, краткое и подробное описание, дату добавления. `Customer` хранит имя, фамилию, email и телефон покупателя. `Order` хранит покупателя, пользователя, дату заказа, статус и адрес доставки. `OrderItem` хранит заказ, товар, количество и цену товара на момент оформления заказа.

Скрипт базы данных в проекте представлен миграциями Django и отдельным файлом `DATABASE_SCRIPT.sql`. Начальная структура таблиц находится в миграции `shop/migrations/0001_initial.py`, а связь заказа с пользователем добавлена в миграции `shop/migrations/0002_order_user.py`. При запуске контейнера команда `python manage.py migrate --noinput` автоматически создаёт все таблицы в PostgreSQL.

Часть реализации программы находится в файле `football_shop/models.py`, где описаны модели базы данных и связи между ними. CRUD-страницы реализованы через generic views в `football_shop/views.py`, API реализовано через Django REST Framework в папке `shop_api`, работа корзины через сессии вынесена в `football_shop/cart.py`, а читаемое отображение таблиц в административной панели настроено в `football_shop/admin.py`.

Скриншот 12 — фрагмент кода `shop/models.py` с моделями `Product`, `Order` и `OrderItem`. Он показывает, как в коде описаны связи между товарами, заказами и пользователями.

Скриншот 13 — фрагмент кода `shop/admin.py`. На нём нужно показать `list_display`, `search_fields`, `list_filter` и inline для позиций заказа. Это подтверждает, что административная панель настроена в читаемом виде.

Часть работы программы можно показать через главную страницу, каталог товаров, корзину, оформление заказа, API или административную панель. Для итогового отчёта лучше использовать скриншоты главной страницы, админки, Docker-контейнеров и логов запуска, потому что они напрямую подтверждают release-настройку проекта.

В результате выполнения итоговой практической работы проект был подготовлен к запуску в Docker. Была настроена структура release-версии, добавлены Dockerfile и docker-compose, настроен PostgreSQL, Gunicorn и Nginx, добавлена сборка статических файлов и автоматическое применение миграций. Работоспособность проекта была проверена через Nginx: главная страница открывается, административная панель доступна, вход администратора работает, а модели приложения отображаются корректно. Таким образом, проект можно запускать как локально для разработки, так и в контейнерах для демонстрации итоговой версии.

Для запуска проекта используется команда:

```bash
docker compose up --build -d
```

Для проверки контейнеров используется команда:

```bash
docker compose ps
```

Для просмотра логов Django/Gunicorn используется команда:

```bash
docker compose logs web
```

Для остановки контейнеров используется команда:

```bash
docker compose down
```
